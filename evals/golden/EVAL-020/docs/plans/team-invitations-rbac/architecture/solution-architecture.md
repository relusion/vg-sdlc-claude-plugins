## Executive Summary

Team invitations and role-based authorization remain in one Python application runtime with explicit authorization, invitation, persistence, trust, and transaction boundaries.

## Scope and Non-Goals

**Scope**

- Role and authorization foundation
- Invitation creation, listing, and acceptance
- Architecture-level component, deployment, data, trust, contract, transition, quality, and operations consequences

**Non-goals**

- Email delivery implementation
- Production deployment topology selection
- Endpoint payload, database schema, or implementation-test specification

## Architecture Overview

A browser or API client calls the invitation component over HTTPS. The invitation component crosses the in-process authorization boundary before PostgreSQL persistence, and invitation acceptance coordinates token consumption with unique membership creation in one transaction.
