## Executive Summary

The existing Python application remains one deployable runtime with explicit logical
boundaries for invitation handling and authorization. A browser or API client calls the
application over HTTPS; the application authorizes every protected operation and commits
invitation and membership state to PostgreSQL. This shape implements the two planned
features without adding a service boundary or changing plan-owned obligations.

## Scope and Non-Goals

In scope are the cross-feature runtime responsibilities, their relationships, the verified
Docker Compose deployment, durable-data ownership, security and interaction re-projections,
and three source-backed quality scenarios. Email delivery, custom roles, cloud-region
selection, file/class design, endpoint payload schemas, test cases, release approval, and
deployment execution remain outside this architecture baseline.

## Architecture Overview

`C-001` is the client-facing UI/API consumer, `C-002` owns the invitation request
workflow, `C-003` owns authorization and membership policy, and `C-004` is the source of
truth for invitation and membership state. `C-002` and `C-003` are logical components in
the same application deployment (`N-002`), consistent with ADR-0001. `C-004` runs on the
separate PostgreSQL node `N-003`.
